1. ¿Cuál es el bug raíz y en qué archivo/función está?
La raíz del bug está en el espacio de usuario de la API de criptografía AEAD en `kernel/linux/crypto/algif_aead.c`, concretamente en el flujo de recepción de datos en la función `_aead_recvmsg()` y en el uso de la copia in-place a través de `crypto_aead_copy_sgl()`. El kernel intenta mezclar asociado data (AAD) con texto cifrado o texto plano en un único buffer de salida, pero asume de forma incorrecta que la longitud de destino coincide con la suma `assoclen + cryptlen`. Esa suposición se rompe cuando los segmentos SGL o los búferes de usuario son más pequeños de lo esperado o cuando se realiza la operación de descifrado in-place a partir de un socket AF_ALG.

2. ¿Por qué el write a dst[assoclen + cryptlen] es peligroso? 
El acceso `dst[assoclen + cryptlen]` es peligroso porque apunta justo fuera del área válida de los datos procesados. En AEAD, `assoclen` describe la longitud de los datos autenticados que no se cifran y `cryptlen` describe la longitud de los datos cifrados/descifrados. Si el búfer `dst` no tiene exacta o suficientemente esa capacidad, la escritura puede sobrescribir memoria adyacente, corrupta el contenido del page cache compartido o incluso afectar páginas de otro inode. En código de bajo nivel del kernel, una escritura fuera de límites puede corromper páginas de caché del archivo y dejar datos inconsistentes sin que el proceso usuario vea un fallo inmediato.

3. ¿Por qué el exploit es "stealthy" (no modifica el archivo en disco)?
El exploit es “stealthy” porque no necesita modificar el archivo en disco. La vulnerabilidad se aprovecha sobre el page cache: el kernel mantiene una copia en memoria de los bloques de un inode y las operaciones de lectura/escritura comunes suelen pasar por esa caché antes de propagarse al disco. Al corromper el page cache asociado a un archivo privilegiado, el exploit puede introducir bytes maliciosos que serán usados cuando otro proceso ejecute o lea el archivo, aunque el respaldo en disco permanezca intacto. Esto hace que la explotación no deje huellas obvias en el sistema de archivos ni cambie el `inode` ni los metadatos `chmod` del archivo en disco.

4. Conecta esto con lo que vimos en clase: page cache, chmod, setuid, inodos
En clase vimos que un inode es la estructura que describe el archivo: propietario, permisos, tipo y punteros a bloques. `chmod` cambia los permisos del inode, y `setuid` activa la elevación de privilegios cuando se ejecuta un binario con ese bit. En este caso, la vulnerabilidad no altera permisos ni el inode en sí; corrompe la memoria del page cache para un binario que ya tiene `setuid root`. Así, el exploit puede conseguir ejecución con privilegios elevados porque un binario confiable cargado desde cache se vuelve malicioso en memoria, aunque el inode siga declarando los mismos permisos y el disco siga conteniendo el mismo contenido.

5. ¿Qué aprendiste sobre cómo múltiples cambios "razonables" pueden crear un bug grave?
Esta falla muestra que múltiples cambios razonables pueden crear un bug grave. Cada uno parece aceptable por separado: usar AEAD para operaciones in-place es una optimización legítima, separar AAD y datos cifrados es normal, y permitir que el kernel maneje scatter-gather list (SGL) en sockets es práctico. Pero cuando la lógica mezcla longitudes (`assoclen + cryptlen`) y la copia de datos se realiza sin reparar en todas las variantes de tamaño y en los límites de buffers, surge una condición de escritura fuera de rango. Es un ejemplo clásico de cómo un conjunto de decisiones aparentemente inocuas —optimizar para evitar copias extra, confiar en que la forma del SGL es siempre correcta, asumir que la suma de longitudes encaja— se combinan para generar un vector de ataque serio.
Aprendí que no basta con que cada función sea razonable en su ámbito; es necesario verificar cómo interaccionan los límites de buffers, las longitudes y la semántica de caché. El bug se esconde en un camino de datos complejo entre la API de sockets AF_ALG, la gestión de SGL y el page cache, y la consecuencia es mayor cuando involucra `setuid` y ejecutables privilegiados. Por eso el análisis de seguridad debe considerar no solo la validez local del código, sino también cómo se propaga la memoria compartida del kernel y cómo se puede abusar de las suposiciones de tamaño.
